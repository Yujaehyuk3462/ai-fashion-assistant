## 2. 에이전트 상태 지속성 + 진단-수리 자기평가 루프 (`9b4252f`)

**상태 지속성**
- 백그라운드 파이프라인(속성 추출, 추천 생성)이 실패하면 그 기회가 조용히 사라지던 문제를 해결.
- `AgentTask` 모델 + `users/{uid}/agent_tasks` 서브컬렉션에 실패한 작업을 재시도 가능한 태스크로 저장.
- `AgentSweeper`가 앱 실행 시 미해결 태스크를 발견해 재개 (실행당 최대 2건, 지수 백오프, 5회 재시도 후 포기).
- 옷장 컬렉션 동시성 가드: `extractionAttemptedAt` 필드로 중복 시도 방지.
- Firestore 규칙에 `agent_tasks` 추가, 사용자 승인 후 배포 완료.
- 앱 실행당 Gemini 호출 총량(AgentSweeper + AgentPlanner.runProactiveCheck 순차 실행 시 최대치)을 계산해 5회 이상이면 폴백 모델까지 감안한 호출 간 텀을 추가.

**진단-수리 자기평가 루프**
- 기존 단순 점수 평가를 "다축 진단 → 약점 축 파악 → 로컬 매칭으로 교체 → 재평가" 패턴으로 업그레이드.
- Gemini 응답을 `[총점]/[격식적합]/[색상조화]/[스타일통일]/[개선점]` 다축 포맷으로 파싱.
- 가장 약한 축을 진단해 해당 아이템만 로컬 매칭으로 교체 후 재평가 (최대 Gemini 3회 호출로 제한).
- 진행 상황을 거의 실시간으로 `agent_logs`에 스트리밍.
- 홈 화면에 "에이전트 작업 중" 미니 인디케이터(점 애니메이션) 추가 — **백그라운드 능동 추천 파이프라인 전용**, 동기적으로 사용자가 기다리는 흐름(AI 코디 분석하기, 주간 플랜)에는 적용하지 않음.

## 6. Gemini 텍스트 폴백 모델 교체: `gemini-2.5-flash` → `gemini-3.1-flash-lite` (`ab31f7d`)

- 배경: 실기기 테스트 중 `gemini-2.5-flash`가 `"no longer available to new users"` 오류로 막혀 있는 것을 발견 — 폴백이 사실상 무력화된 상태였음.
- ListModels API로 `models/gemini-3.1-flash-lite` 존재 확인 후 `generateContent` 실호출로 정상 응답(HTTP 200) 확인.
- 실제 앱 코드의 `GeminiService.withTextModelFallback`을 대상으로, 1차 모델 타임아웃을 인위적으로 재현해 실제로 `gemini-3.1-flash-lite`로 재시도되고 성공하는지 임시 테스트로 엔드투엔드 검증(검증 후 즉시 원복).
- 검증 과정에서 확인한 사실: 존재하지 않는 모델명으로 1차 모델을 깨면 404(재시도 불가 오류)가 나서 폴백이 트리거되지 않음 — `GeminiApiException.isRetryable`은 503/429만 재시도 대상으로 보기 때문(의도된 동작). 그래서 검증은 실제 트리거 조건인 타임아웃으로 재현.

## 7. AI 피팅 로딩 팝업 + 플로팅 진행 아이콘 + 추천 옷 슬라이드 (`0852b2d`)

- 피팅 분석/합성이 진행되는 동안 결과 카드를 접고, 화면 어디서든(탭 이동해도) 진행 상태를 볼 수 있는 드래그 가능한 플로팅 아이콘을 앱 셸(`main.dart`) 레벨에 추가. 진행 중/완료(체크+바운스 애니메이션, 30초 후 자동 숨김)/실패 상태 표시.
- **설계 원칙**: 로딩 팝업은 순수 대기 화면 전용 — 분석 결과·점수·피팅 이미지는 절대 팝업 안에 넣지 않는다. 작업이 끝나면 팝업은 사라지고 결과는 원래 인라인 위치에 그대로(수정 전과 동일하게) 표시된다.
- 팝업 안에는 `OutfitMatcher`로 로컬 매칭한(Gemini 호출 없음) 추천 옷 캐러셀(3초 자동 슬라이드)을 넣어 대기 시간을 활용:
  - 카테고리별 상위 2개 + 전역 상위 5개로 매칭 폭을 넓혀, 피팅 중인 아이템과 같은 카테고리의 다른 아이템도 추천에 포함되도록 함(피팅 itemId만 제외, 카테고리 전체 제외 안 함).
  - `lib/constants/outfit_reason.dart` (신규): 색상 관계 → 격식 일치 → 카테고리별 한 마디 순으로 적용되는 로컬 규칙 기반 문구 생성. 항상 어떤 피팅 아이템과 왜 어울리는지 명시(예: "피팅 중인 블랙 상의와 어울리는 하의예요"). 속성이 부족하면 안전한 기본 문구로 폴백.
  - 이미지 크롭 문제(`BoxFit.cover`)를 `BoxFit.contain` + 배경색으로 수정.
- `lib/services/fitting_progress.dart` (신규): 팝업 접힘 상태를 앱 전역에서 공유하는 `ValueNotifier<bool>`.
- 플로팅 아이콘 드래그: `GestureDetector`에서 `onTap`과 `onPanUpdate`를 함께 배선하면 제스처 아레나에서 탭이 드래그를 가로채는 문제가 있어, `onTap`을 쓰지 않고 `onPanDown/Update/End`로 직접 탭-드래그를 판별하도록 변경(누적 이동 거리가 임계값 이하면 탭, 넘으면 드래그로 처리). 화면 가장자리 안전 여백을 두고 clamp.
- 홈 화면 인사말 하드코딩 이름 수정: `'민준님'` → `'디티오님'`.

## 8. 플로팅 피팅 아이콘 드래그 안 되던 문제 수정 (`382d75e`)

- 7번 항목 배포 후 실기기에서 드래그해도 아이콘이 전혀 움직이지 않는 문제 재발견. 처음엔 제스처 아레나 경쟁을 의심했지만, `onPanDown`/`onPanUpdate`에 디버그 로그를 심어 확인해보니 팬 이벤트 자체는 정상적으로(누적 이동거리 50~100 이상) 계속 들어오고 있었음 — 즉 제스처 인식 문제가 아니었음.
- 실기기 로그에서 `panUpdate`가 찍힐 때마다 `Incorrect use of ParentDataWidget` 예외가 함께 던져지는 것을 발견하며 진짜 원인 확인: `Positioned`를 `LayoutBuilder`의 builder 콜백 **안**에서 반환하고 있었는데, `LayoutBuilder` 자체가 `RenderObjectWidget`이라 `Positioned`가 `Stack`이 아니라 `LayoutBuilder`의 RenderObject에 `StackParentData`를 적용하려다 매 드래그 프레임마다 예외가 발생 — 좌표(`_left`/`_top`)는 정상적으로 갱신되고 있었지만 그 갱신이 화면에 전혀 반영되지 않고 있었음.
- 제약(constraints)을 읽는 `LayoutBuilder`를 `AppShell`의 `Stack` 바깥으로 끌어올리고 `maxWidth`/`maxHeight`를 `_FloatingFittingIcon`에 생성자 파라미터로 전달, `Positioned`가 `Stack`의 (비-RenderObjectWidget 래퍼만 거친) 직계 자손이 되도록 재구성해서 해결.
- 실기기에서 (a) 꾹 눌러 드래그 시 아이콘이 실제로 따라 움직이는 것, (b) 탭 시 기존처럼 피팅룸 이동+팝업 펼침이 되는 것을 사용자가 직접 확인.
